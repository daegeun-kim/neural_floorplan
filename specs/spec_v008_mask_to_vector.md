# Spec v008: Mask-to-Vector Reconstruction for Neural Floorplan

## 0. Purpose

This document defines the first mask-to-vector reconstruction stage for Neural Floorplan.

The goal is to convert a noisy 5-class CNN prediction into a clean SVG preview using the component primitives defined in spec v007.

Pipeline position:

```text
CNN predicted segmentation
â†’ class masks
â†’ mask cleanup
â†’ primitive candidate extraction
â†’ primitive fitting
â†’ SVG preview output
```

## 1. Scope

This spec covers:

```text
- reading CNN prediction images
- decoding 5-class predicted masks
- ignoring icon class
- extracting wall, opening, and room evidence
- fitting clean primitives
- separating door/window candidates heuristically
- generating SVG output
```

This spec does not cover:

```text
- CNN training
- CNN evaluation
- creating new CNN classes
- retraining the model
- room adjacency graph construction
- door topology validation
- final DXF export
- Grasshopper integration
- JSON export
```

## 3. Active Classes

Input classes:

| ID | Class | v008 action |
|---:|---|---|
| 0 | background | ignore |
| 1 | wall | vectorize |
| 2 | opening | vectorize and classify heuristically |
| 3 | room | vectorize for SVG preview |
| 4 | icon | ignore |

Icon pixels should be ignored in v008.

Do not remove the icon class from the CNN inside this stage. Removing it would require a separate CNN retraining decision.

## 4. Door vs Window Strategy

The CNN currently predicts one `opening` class. It does not directly separate doors and windows.

v008 should not retrain the CNN.

Instead, v008 should attempt heuristic opening separation after extracting opening components.

Recommended initial heuristic:

```text
window_candidate:
    long and narrow component
    high aspect ratio
    aligned with host wall
    often located on exterior wall

door_candidate:
    more compact component
    aspect ratio closer to 1
    larger blob/chunk near wall break
    may have swing-like evidence if visible in original/predicted image

generic_opening:
    uncertain classification
```

If classification confidence is low, output a generic opening instead of forcing a door/window decision.

## 5. Scale and Unit Handling

The preferred output is real-world units.

However, a raster prediction alone does not reliably define real-world scale.

v008 should include a scale resolver with this priority:

```text
1. explicit scale metadata if available
2. original SVG/dataset metadata if available
3. dimension text parsed elsewhere if available
4. standard architectural assumptions, such as door width
5. fallback to pixel units
```

For the first implementation, the system may export SVG in pixel coordinates if scale is unknown.

The SVG output should record the unit status:

```text
unit="px"
scale_status="unknown"
```

or:

```text
unit="mm"
scale_status="resolved"
```

Do not silently pretend pixel units are real-world dimensions.

## 7. Main Pipeline

### Step 1 â€” Load prediction image

Input:

```text
sample_000_prediction.png
```

Output:

```text
semantic class map
```

If the image is already class-ID encoded, read class IDs directly.

If the image is color-coded, decode colors into class IDs.

### Step 2 â€” Split class masks

Create binary masks:

```text
wall_mask
opening_mask
room_mask
```

Ignore:

```text
background_mask
icon_mask
```

### Step 3 â€” Clean masks

Apply lightweight cleanup:

```text
remove tiny components
fill small holes
close small gaps
smooth jagged noise
```

Class-specific intent:

```text
wall_mask:
    preserve global wall structure
    close small gaps
    remove isolated blobs

opening_mask:
    preserve opening locations
    remove tiny fragments
    avoid over-smoothing

room_mask:
    preserve broad room regions
    simplify noisy boundaries
```

### Step 4 â€” Extract wall candidates

Recommended first approach:

```text
wall mask
â†’ skeletonization or medial axis
â†’ line detection
â†’ merge collinear segments
â†’ snap near-horizontal and near-vertical lines
â†’ WallPrimitive objects
```

Internal wall representation should use:

```text
centerline + thickness
```

Reason:

```text
- easier to host openings
- easier to snap and merge
- less sensitive to noisy wall boundaries
- more CAD-like than raw contours
```

### Step 5 â€” Extract opening candidates

Use connected components from `opening_mask`.

For each component, compute:

```text
bbox
area
aspect_ratio
major_axis
center_point
nearest wall candidates
alignment with wall
```

Then attach to nearest suitable wall.

An opening without a host wall should be marked unresolved in the debug SVG.

### Step 6 â€” Classify opening candidates

After host-wall assignment, classify each opening as:

```text
door_candidate
window_candidate
generic_opening
```

Use geometry-based heuristics only.

Do not use a new neural network in v008.

Suggested heuristic inputs:

```text
component aspect ratio
component area
component width along wall
component thickness across wall
distance to exterior boundary
host wall type if known
shape compactness
```

The first implementation should keep thresholds configurable.

Example config:

```yaml
opening_classification:
  window_min_aspect_ratio: 2.5
  door_max_aspect_ratio: 2.0
  min_confidence_for_type: 0.65
```

### Step 7 â€” Generate primitives

Convert candidates into v007 primitives:

```text
wall candidates       â†’ WallPrimitive
opening candidates    â†’ OpeningPrimitive
window candidates     â†’ WindowPrimitive
confident door candidates â†’ DoorPrimitive
room components       â†’ RoomPrimitive
```

If door classification is uncertain, generate `OpeningPrimitive` only.

### Step 8 â€” Generate room polygons

Use room mask components to create simplified room polygons.

Room processing should remain simple in v008:

```text
room mask
â†’ connected components
â†’ contours
â†’ polygon simplification
â†’ SVG room polygons
```

Do not enforce room adjacency graph or room-to-door topology in v008.

### Step 9 â€” Apply basic geometric rules

v008 should include only basic geometric rules:

```text
straighten wall lines
snap near-horizontal and near-vertical walls
merge nearby collinear wall segments
remove tiny wall fragments
attach openings to nearest wall
project openings onto host wall
```

Exclude full topology validation:

```text
room adjacency graph
circulation correctness
door connects two rooms
complete room closure solving
```

These are later stages.

### Step 10 â€” Export SVG

Generate a clean SVG preview.

SVG should show:

```text
rooms as light polygons
walls as clean strokes or wall polygons
openings as hosted gaps/segments
doors as clean primitive symbols when confidently classified
windows as clean hosted linear symbols when confidently classified
unresolved objects in debug layer
```

## 10. Door Symbol Rule

v008 may generate a simple hinged door symbol only when an opening is confidently classified as a door.

Door typology variants are excluded.

Allowed:

```text
single hinged door primitive
```

Excluded:

```text
sliding door
double door
folding door
custom door types
```

If swing direction is unknown, choose a default direction and mark the door as low confidence in the debug layer.

## 11. Window Symbol Rule

v008 may generate a simple linear window symbol when an opening is confidently classified as a window.

Window variants are excluded.

Allowed:

```text
generic wall-hosted window line/block
```

Excluded:

```text
bay window
curtain wall
sliding window variants
window family classification
```

## 12. Icon Rule

Ignore icon pixels in v008.

The icon class should not block wall, opening, or room vectorization.

If icon noise interferes with masks, it may be removed during cleanup.

Optional quantitative checks:

## 19. Topology-Aware Vectorization (task05)

### Goal

Move from pixel-contour tracing to spatially generalised, topology-aware primitives. CNN segmentation output is used as evidence, not as a pixel-perfect boundary.

### Floor / Building Footprint

- Floor is derived from the union of `wall + opening + room + icon` foreground masks.
- Implemented in `src/vectorization/floor_extraction.py`:
  - `_rectilinearize_contour(pts)` â€” classifies each contour segment as H or V, cascades snapping, and closes the polygon so the wrap-around edge is also axis-aligned.
  - `extract_floor(masks, scale_info, dilate_px, simplify_epsilon_ratio)` â€” dilates/closes the union mask, finds the largest contour, simplifies it, and calls `_rectilinearize_contour`.
- `FloorPrimitive` defined in `src/vectorization/primitives/floor.py` â€” renders a filled `<polygon>` element.
- Floor appears first in the SVG (behind all other geometry).
- `split_class_masks()` now returns an `"icon"` key in addition to wall/opening/room.

### Door Primitive (3-component parametric symbol)

Each door contains exactly three SVG elements:

1. **Origin segment** â€” along the wall (hinge â†’ leaf end)
2. **Opening segment** â€” perpendicular to the wall, same length (hinge â†’ panel end)
3. **Swing arc** â€” quarter arc from origin end to panel end, radius = width, centre = hinge

Implemented in `src/vectorization/primitives/door.py`:
- `_leaf_end()` â€” tip of closed door (along wall direction, sign determined by swing side)
- `_panel_end()` â€” tip of open door (perpendicular to wall; `angle + 90Â°` for "left", `angle âˆ’ 90Â°` for "right")

**Swing direction** chosen by probing `room_mask` pixels in each perpendicular direction from the opening centre (`_pick_swing_direction` in `src/vectorization/opening_classification.py`). Falls back to `"left"` when room evidence is tied or absent. `classify_openings()` now accepts `room_mask` parameter.

### Wall Splitting at Openings

Implemented in `src/vectorization/geometry_rules.py` â†’ `split_walls_at_openings(walls, hosted, min_segment_px)`:

1. Project each hosted opening/window center Â± width/2 onto the host wall (parametric t-values).
2. Sort and merge overlapping gap intervals.
3. Output is the complement: solid wall segments outside every gap.
4. Walls with no hosted openings are returned unchanged.

Called in `process_single` after `apply_geometry_rules`, on windows and hosted-unresolved openings.

### SVG Layer Order

Back-to-front (as rendered):

```
floor â†’ rooms â†’ walls â†’ openings â†’ windows â†’ doors â†’ debug
```

### Integration

`src/vectorization/run_mask_to_vector.py` `process_single`:

1. Calls `classify_openings(..., room_mask=room_mask)` for swing direction.
2. Calls `split_walls_at_openings(walls, hosted_for_split)` after geometry rules.
3. Calls `extract_floor({wall, opening, room, icon masks})` to build the floor footprint.
4. Passes `floor=floor` to `build_svg`.

## 20. Four Final Vector Classes â€” Floor / Wall / Opening / Icon Only (task06)

### Goal

Supersede the room-as-separate-vector-class direction from task05. CNN class
`3 - room` is now treated purely as **floor evidence**; the final SVG exports
**exactly four** singular-named classes in this group order:

```
<g id="floor"> <g id="wall"> <g id="opening"> <g id="icon">
```

produced procedurally in pipeline order `wall â†’ floor â†’ opening â†’ icon`.
Background (class 0) is ignored as before.

This section supersedes:
- Â§6 "SVG should use separate layers/groups" (`rooms`/`walls`/`openings`/`doors`/`windows`/`debug`) â†’ now `floor`/`wall`/`opening`/`icon` (+ debug-only, non-final).
- Â§8 module structure listing `room_extraction.py` and `primitives/room.py` as pipeline modules â†’ both removed from the pipeline (see below).
- Â§9 example config's `svg:` block and `draw_rooms`/`draw_doors`/`draw_windows` keys â†’ replaced by `draw_floor`/`draw_wall`/`draw_opening`/`draw_icon`.
- Â§19 "SVG Layer Order" (`floor â†’ rooms â†’ walls â†’ openings â†’ windows â†’ doors â†’ debug`) â†’ now `floor â†’ wall â†’ opening (doors+windows) â†’ icon â†’ debug`.

### Wall: Outer Rectilinear Loop, Then Inner Walls

`src/vectorization/wall_extraction.py` is now a two-stage, contour-driven extractor:

1. **`extract_outer_wall_loop(wall_mask, room_mask=None, ...)`** â€” unions wall + room evidence, closes/dilates to bridge gaps, takes the largest external contour, simplifies it, and rectilinearizes it with `_rectilinearize_contour` (moved here from `floor_extraction.py`). The polygon vertices become a closed ring of axis-aligned `WallPrimitive` centerline segments (`wall_outer_NNNN`), guaranteed rectilinear by construction. Returns `(outer_walls, outer_polygon)`.
2. **`extract_inner_walls(wall_mask, outer_polygon, thickness, ...)`** â€” erases a band around the outer polygon from `wall_mask` (`_erase_outer_band`), then runs the existing skeletonize + `HoughLinesP` + collinear-merge pipeline on the remainder. Returns `inner_walls` (`wall_inner_NNNN`).
3. **`extract_walls(wall_mask, room_mask=None, ...)`** â€” thin orchestrator: `(outer_walls, inner_walls, outer_polygon) = extract_walls(...)`. Outer loop is always computed before inner walls.

Wall angle snapping moved to `geometry_rules.snap_walls_to_45` (replaces `snap_walls_to_cardinal`): walls within `snap_threshold_deg` of a cardinal (0/90) snap to that cardinal (orthogonal preference); all other walls snap to the nearest 45Â° multiple, so every wall reads as architectural linework.

### Floor: Direct Translation of the Outer Wall Loop

`floor_extraction.py` no longer unions all four class masks. `extract_floor(outer_polygon, scale_info=None)` is now a thin wrapper that returns `FloorPrimitive(polygon=outer_polygon)` directly â€” the floor boundary *is* the outer wall loop, filled with `stroke="none"`.

### Opening: Project Before Classify, Then Split

Pipeline order in `run_mask_to_vector.process_single` was corrected so opening centers are projected onto the (already 45Â°-snapped) host wall centerline **before** door/window classification â€” previously classification ran first, so door hinge points could be computed from an unprojected center. Order is now:

```
extract_openings â†’ apply_geometry_rules (snap walls + project openings) â†’ classify_openings â†’ split_walls_at_openings
```

`classify_openings` gained an optional `walls` parameter so `WindowPrimitive` can inherit the host wall's `thickness` instead of a hardcoded fallback.

Doors and windows are **not** separate top-level SVG classes â€” both render inside the single `<g id="opening">` group. Unresolved/floating openings remain debug-only (`<g id="debug">`), never in `opening`.

### Window Primitive Simplified

`primitives/window.py`: replaced the polygon+midline sill box with a single `<line>` styled like a wall centerline (`stroke="#3355cc"`, `stroke-width=thickness`) â€” "blue wall-aligned line segment" per the task. `WindowPrimitive` gained a `thickness` field.

### New IconPrimitive

`primitives/icon.py` (new) + `icon_extraction.py` (new): `extract_icons(icon_mask, min_area, scale_info)` follows the same connected-components â†’ largest contour â†’ `approxPolyDP` simplify pattern as the old room extraction, producing filled `IconPrimitive` shapes (`fill="#4caf50"`). `cleanup.clean_icon_mask` added alongside the existing wall/opening/room cleanup functions.

### RoomPrimitive / room_extraction Removed From the Pipeline

`room_extraction.py` was deleted (floor and door-swing-direction logic only need the raw `room_mask`, not room polygons). `primitives/room.py` (`RoomPrimitive`) is kept as a standalone primitive â€” still exported from `src/vectorization/primitives/__init__.py` and covered by `tests/test_vectorization_v007.py` â€” but it is never imported by `run_mask_to_vector.py`, `export_svg.py`, or `floor_extraction.py`, so it never appears in final SVG output.
