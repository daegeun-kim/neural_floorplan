# Spec v005: Raster-To-Graph Phase 4

## 0. Purpose

This spec defines Phase 4 vectorization: using Raster-to-Graph to predict a simple orthogonal wall graph from raster input.

The motivation is that rule-based vectorization from segmented masks has reached a practical limit. The required spatial logic for wall alignment, junction detection, and graph connectivity is too high for simple CV component extraction and hand-written rules alone.

Current status:

```txt
Phase 4 is currently settled as pretrained Raster-to-Graph inference.
No fine-tuning is needed for the current project version.
```

Current active target:

```txt
model_clean.png raster -> simple orthogonal wall graph
```

Older fine-tuning notes in this file are retained as historical fallback planning only. The current implementation details are also tracked in `spec_v010_raster2graph_modifications.md`.

## 1. Phase 4 Definition

Phase 4 keeps the wall-graph goal from the existing SVG/mask pipeline, but changes the practical method to pretrained Raster-to-Graph inference from the clean `model_clean.png` raster.

Pipeline:

```txt
-> model_clean.png
-> crop content bbox
-> add true 20% white padding
-> scale long edge to 512 px on white canvas
-> pretrained Raster-to-Graph checkpoint inference
-> generous graph generation and validity scoring
-> mask-and-rerun multistart recovery
-> merge-on-intersection and light cleanup
-> predicted wall graph
-> later CAD output
```

This phase is not a continuation of point-based OpenCV vectorization. It is a graph-prediction phase using the external Raster-to-Graph checkpoint as the main wall graph extractor. Fine-tuning is deferred because the current inference pipeline is satisfactory.

## 3. Input Raster Content

The current inference version makes `model_clean.png` as compatible as practical with the original Raster-to-Graph repository's image requirements while adding standardized margin:

```txt
RGB floor-plan raster
white background
content cropped to dark bbox
true 20% white padding around content
longer padded image edge scaled to 512 px
paste centered on a 512x512 white canvas
normalize with the original repo mean/std
```

The preprocessing keeps the original repo canvas and normalization, with local true-padding before resizing:

```txt
crop content bbox from model_clean.png
add standardized_margin = 0.20 on all sides as real white pixels
scale_factor = 512 / max(padded_width, padded_height)
canvas = 512x512 white RGB
paste resized plan centered on canvas
mean = [0.920, 0.913, 0.891]
std  = [0.214, 0.216, 0.228]
```

The current Phase 4 path uses only `model_clean.png`. Do not mix in `F1_scaled.png` / original messy rasters for the settled wall graph inference method.

Rationale:

```txt
model_clean.png isolates graph learning from scan/listing noise
F1_scaled.png introduces style, clutter, contrast, and resolution variation
mixing both makes failure analysis harder
```

The stored artifact may include a per-sample preprocessed copy for reproducibility, but the source-of-truth input path remains the sample's `model_clean.png`.

## 6. Model Goal

The current inference stage should predict a simple wall graph.

Minimum output:

```txt
wall junction/end nodes
wall segment edges
```

Opening-center annotation nodes are deferred. If they are revived later, they should not be wall edge endpoints; they should lie on a predicted or ground-truth wall edge and indicate whether the opening is a door or a window.

No wall thickness, room, furniture, fixture, or detailed semantic attributes are required in v1.

This is intentionally simpler than the full Raster-to-Graph paper.

## 7. Current Model Architecture

Use the external Raster-to-Graph model/checkpoint as the current inference base.

The previous SegFormer-backbone heatmap/edge-head baseline and Raster-to-Graph fine-tuning plan remain fallback ideas, but the current direction is inference with the official-style Raster-to-Graph architecture already copied under `external/raster_to_graph/`.

Current architecture:

```txt
model_clean.png raster input
-> crop512_margin20_truepad preprocessing
-> ResNet-50 backbone from checkpoint
-> Deformable-DETR / autoregressive graph decoder
-> point, edge, this-edge, last-edge, semantic predictions
-> graph assembly
-> local thresholding, reranking, multistart recovery, merge, and cleanup
-> graph_pred.json / graph_pred.svg
```

The model is currently used as pretrained graph prediction, not semantic mask prediction.

Historical fallback fine-tuning schedule:

```txt
stage 1: load checkpoint0299.pth and freeze the ResNet backbone
stage 1: train graph decoder / prediction heads on CubiCasa model_clean.png + wall_graph.json labels
stage 2: unfreeze later ResNet stages if predictions stay underconfident or empty
stage 3: optionally unfreeze more backbone layers only after graph decoder adaptation is stable
use a lower learning rate for backbone than decoder/heads
do not keep autoregressive graph parts frozen for long, because the graph grammar must adapt
```

Recommended learning-rate starting points:

```txt
backbone frozen stage: decoder/heads lr = 1e-4 to 2e-4
partial unfreeze stage: backbone lr = 1e-5 or lower
partial unfreeze stage: decoder/heads lr = 1e-4 to 2e-4
use checkpoint validation metrics to decide when to unfreeze
```

If training becomes necessary later, this schedule would keep useful pretrained graph-decoding behavior while adapting both the visual domain and graph grammar from:

```txt
original dataset_v5 floor-plan style -> original graph format
```

to:

```txt
CubiCasa model_clean.png -> this project's wall_graph.json-derived target
```

## 8. Alternative Model Approaches

Historical/fallback model options:

### Option A - SegFormer Backbone Plus Heatmap And Edge Heads

```txt
model_clean.png raster
-> SegFormer encoder/backbone
-> node heatmaps
-> opening-center heatmaps
-> node extraction
-> candidate edge classifier between node pairs
-> graph cleanup
```

This was the earlier baseline implementation. It is simpler than full autoregressive graph generation and remains a fallback if the current pretrained inference path stops being viable.

### Option B - DETR-Style Node/Edge Proposal

```txt
input raster
-> transformer queries
-> node coordinates
-> directional connection attributes
-> graph construction
```

This is closer to Raster-to-Graph.

### Option C - Raster-to-Graph Inspired Autoregressive Model

```txt
input raster + partial graph mask
-> predict next graph nodes and connections
-> repeat until graph complete
```

This is the most similar to Sizhe Hu's Raster-to-Graph paper. Because the official-style code and checkpoint are now available under `external/raster_to_graph/`, this is the current architecture family, but used through inference rather than fine-tuning.

Current implementation:

```txt
pretrained checkpoint0299.pth inference with local preprocessing, thresholds, scoring, multistart, and merge/filter cleanup
```

Do not start with full semantic room prediction.

## 9. Training Target Simplification

The graph target should ignore:

```txt
wall thickness
room categories
furniture
fixtures
text
icons
```

The model should focus on:

```txt
where wall graph nodes are
which nodes are connected by orthogonal wall edges
where door centers are located
where window centers are located
which wall edge hosts each opening center when this can be derived reliably
```

Door and window centers should be learned as distinct annotation node classes, not as generic opening nodes. Detailed door swing geometry, window span geometry, and CAD dimensions can still be attached later using the existing seven-class masks and CAD rules.

## 10. Evaluation Metrics

Use graph-level metrics inspired by Raster-to-Graph:

### Node Accuracy

A predicted node is correct if it is within a configured pixel threshold of a ground-truth node.

Recommended first threshold:

```txt
5 px
```

### Edge Accuracy

A predicted edge is correct if both endpoint nodes match the corresponding ground-truth edge endpoints within threshold.

### Structure Accuracy

A sample has correct structure only if all required nodes and edges are matched and no major extra edges exist.

### Opening Center Accuracy

A predicted opening-center node is correct if:

```txt
its type matches door_center or window_center
its coordinate is within threshold of the ground-truth opening center
it lies on or near the correct host wall edge when host-edge labels are available
```

### Orthogonality

All predicted edges must be horizontal or vertical.

### Visual Debug

Generate a graph overlay for every validation sample:

```txt
input raster
ground-truth graph
predicted graph
error overlay
```

## 14. Historical Implementation Notes

The notes below document earlier SegFormer-based and fine-tuning-oriented baselines. They remain for history and fallback planning, but the current implementation target is the pretrained Raster-to-Graph inference path described above and in `spec_v010_raster2graph_modifications.md`.

The earlier local fine-tuning package (`src/raster2graph/`, `src/train_raster2graph.py`, `configs/train_raster2graph_b0.yaml`, `configs/train_raster2graph_finetune.yaml`, and `tests/test_raster2graph.py`) has been removed from the active repo cleanup because Phase 4 no longer needs model training. If fine-tuning is revived later, this section can be used as historical guidance rather than as a current file inventory.

### Architecture (spec SS7/SS8 Option A)

`GraphDecoder` copies `src.models.FloorplanDecoder`'s multi-scale fusion block (per-stage
1x1 projection -> upsample to stage-0 resolution -> sum -> Conv/BN/GELU hidden layers), but
stops at the fused feature map (e.g. H/4 resolution) instead of upsampling to full input
resolution, and ends in 3 sigmoid node-heatmap heads (`wall_node`/`door_center`/
`window_center`) instead of a semantic classifier — standard for keypoint-heatmap heads.
`EdgeClassifier` is a small MLP over pairs of points bilinearly sampled (`F.grid_sample`)
from that same fused feature map, concatenated with their normalized offset/distance — the
"candidate wall-edge classifier" head SS7 asks for. It operates on one sample at a time
(no batch dimension) since each sample has a different node/edge count.

### Host-edge classifier: deferred (judgment call)

SS7 lists a host-edge classifier for opening centers as "optional." v1 does not train one —
at evaluation/inference time, an opening center's host edge is assigned geometrically
(nearest final wall edge by perpendicular distance), the same approach already implemented
and tested in `generate_wall_graphs.py`'s `_host_opening`/`_point_to_wall_distance`. A
trained host-edge head can be added later without changing the node/edge heads.

### Training targets: `wall_graph.json`, not `wall_graph_debug.svg` (judgment call)

The manifest's `graph` field points at `wall_graph_debug.svg` (matching SS5's literal
schema), but `graph_targets.py` actually parses the sibling `wall_graph.json` for numeric
node/edge targets — same data, already machine-readable, with no need to re-derive
coordinates from SVG markup that was only ever formatted for visualization (the debug SVG
embeds its background as a base64 PNG and isn't meant to be re-parsed).

### Heatmap targets are in a separate coordinate space from the loss/metric threshold (judgment call)

Ground-truth node coordinates are rescaled per-axis from the label's native raster size
into `[0, image_size)` (matching the non-aspect-preserving square resize the input image
itself goes through). Heatmap targets are then rendered at a *further* reduced
`heatmap_size` (the fusion stride, e.g. 128 for a 512 input) — `graph_targets.render_heatmap`
handles that second scaling, and `metrics.extract_peaks` inverts it when reading predictions
back out. spec SS10's "5 px" node-accuracy threshold is applied in the `image_size` space
(where ground truth and extracted predictions both live), not the heatmap's own coarser
grid or the original native raster resolution.

### Manifest key adapter for feature caching (judgment call)

`src.feature_cache.extract_features_for_split` (reused verbatim — it only ever caches
backbone hidden states) expects `image`/`target` keys, while the SS5 manifest schema
mandates `input`/`graph` key names. `dataset.manifest_to_feature_cache_index` adapts one to
the other via a temp index file, the same "write a temp adapted index" pattern
`train_segmentation.py` already uses for its `--overfit` mode.

### Structure accuracy is strict (judgment call)

SS10 defines structure accuracy as "all required nodes and edges are matched and no major
extra edges exist" without defining "major." v1 reads this strictly: zero unmatched nodes
on both sides and 100% edge recall/precision, via `metrics.structure_accuracy`.

### Heatmap-head bias initialization (bug found during first real training run)

The first real (non-smoke) training run showed `val_wall_node_recall`/`edge_recall` stuck
at exactly 0 for many consecutive epochs while the loss decreased smoothly — direct
inspection of predicted heatmaps (not just the loss number) showed the model had collapsed
to predicting near-zero everywhere, never forming a real peak even at true node locations.
Root cause: `GraphDecoder`'s heatmap head `Conv2d` layers used PyTorch's default zero bias,
so training starts predicting ~0.5 at every pixel; with ~16,000 background pixels against
only ~15-30 true positive pixels per heatmap, early training overshoots into suppressing
everything before the much sparser positive gradient can recover real peaks — a known
focal-loss failure mode, normally avoided by initializing the head's bias to a negative
"background prior" (RetinaNet/CenterNet convention). Fixed by initializing all 3 heads'
bias to `-4.0`. Also switched `checkpoint.monitor` from `val_structure_accuracy` (an
all-or-nothing metric that would likely stay at 0 for most/all of a run regardless of model
quality) to `val_wall_node_recall`, so `best.pt` tracks meaningful progress. As of this
writing the bias fix alone did not fully resolve the collapse on a restarted real run either
(re-inspecting predictions at the true peak pixel still showed near-zero values several
epochs in) — training was paused for further debugging before completing a full run. No
trained checkpoint from this spec has been validated as producing real graphs yet.

## 15. Pretrained External Checkpoint Test (task21)

A pretrained checkpoint (`checkpoints_Raster2Graph/checkpoint0299.pth`) advertised as
SizheHu's "Raster-to-Graph" model (the architecture referenced in SS8 Option C) was tested
for inference-only use, without fine-tuning. Findings, full detail in
`outputs/raster2graph/pretrained_sizhehu_test/notes.md`:

- The checkpoint is a Deformable-DETR-based set-prediction model (ResNet-50 backbone, 500
  learned queries, 6-layer deformable-attention decoder, plus point/edge/"this-edge"/
  "last-edge"/4-way quadrant-semantic prediction heads) trained on the paper's own
  `dataset_v5`, not CubiCasa5K — confirmed via the checkpoint's embedded `args` and its
  `model` state-dict's module names (`sampling_offsets`/`reference_points` are
  Deformable-DETR's `MSDeformAttn` signature).
- None of the required code existed in this repo initially (model class, the compiled
  `MSDeformAttn` CUDA extension, input preprocessing, query-to-graph decoding, SVG export).
  The checkpoint was **not** force-loaded into the current SegFormer architecture.
- This project's own preview PNGs (`outputs/raster2graph/runs/.../previews/`) turned out to
  be blank diagnostic drawings, not real rasters — traced back to the real
  `masks/debug_overlay.png` inputs instead (4 samples copied into
  `outputs/raster2graph/pretrained_sizhehu_test/inputs/`).

**Follow-up — inference actually run.** The user cloned SizheHu's official repo outside
this project and asked for the minimal code needed to run this one checkpoint to be copied
in. That now lives in `external/raster_to_graph/`, with 3 small patches (none touching
model weights): the compiled `MSDeformAttn` CUDA op was swapped for the repo's own
documented pure-PyTorch fallback (`F.grid_sample`-based; valid since inference has no
backward pass and this environment has no `nvcc`/CUDA toolkit to compile against),
`util/misc.py`'s now-dead torchvision<0.7 compat shim was removed, and 2 removed/changed
APIs (`Image.ANTIALIAS`, `torchvision.transforms.functional`) in `dataset_demo.py` were
replaced with direct equivalents. The checkpoint then loaded with `strict=True` — all 455
tensors matched exactly, confirming the reconstructed architecture is correct.

Result on the 4 traced-back real samples: 1 of 4 produced a real graph (13 points, 14
edges, visually plausible orthogonal partial structure) — but that sample's
`debug_overlay.png` turned out to contain two floor-plan units side by side (a real
CubiCasa5K quirk), which the model's fixed-512-canvas preprocessing squeezed into one
out-of-distribution input, so this isn't a clean success. The other 3 (normal single-unit
images) produced zero detections at the model's default 0.5 confidence threshold. Full
detail, per-sample results, and the updated evaluation-question answers are in
`outputs/raster2graph/pretrained_sizhehu_test/notes.md`.

Conclusion: getting inference running was itself substantial (CUDA-extension workaround,
3 environment/API patches) but turned out tractable in-session, unlike the earlier
estimate. Output quality is inconclusive from 4 samples (1 partial success on a
non-representative image, 3 empty) — a larger-scale test is needed before deciding whether
this pretrained model is a viable path versus continuing to debug/train the SegFormer
baseline.

## SS20 — Task 24: Recovery Inference

**Objective:** Test whether partially generated graphs can be improved by retrying failed
autoregressive steps rather than stopping immediately.

**Script:** `external/raster_to_graph/run_inference_recovery.py` (`--max-samples N`)

- Per setting: `input.png, graph_pred.svg, graph_overlay.png, graph_overlay_rescue_debug.png, metrics.json, notes.txt`

**Recovery strategies:**
1. **Reserve candidates** — pool of low-score (0.05…threshold) candidates as step fallback
2. **Early-end suppression** — ignore end token until `min_nodes=8` generated
3. **Edge retry** — retry graph assembly with wider tolerance (12px) if normal (8px) fails
4. **Step cascade** — ordered retry: main+base_tol → main+retry_tol → reserve+base_tol → reserve+retry_tol
5. **Component restart** (optional) — start a new disconnected component after partial stop

## SS21 — Task 25: Multi-Start Disconnected Component Inference

**Objective:** Generate multiple disconnected graph components from one image: after the initial
component stops, search for a far-away valid start candidate and seed a new component, repeating
up to `max_new_starts=4` times.

**Script:** `external/raster_to_graph/run_inference_multistart.py` (`--max-samples N`)

**Input:** `crop512_margin05/input.png` only (best Task 22 variant).

**Output:** `outputs/vectorization/phase4_raster2graph_multistart_inference/run001/<sid>/<setting>/`
- Per setting: `input.png, graph_pred.svg, graph_overlay.png, graph_overlay_components.png, metrics.json, components.json, notes.txt`

**Core algorithm:**
1. Run tuned base inference (Task 23 best setting) → component 0
2. Build covered-region mask: nodes as 16px disks + edges as 32px lines + 12px dilation
3. Query model on current tensor state; filter candidates: score≥threshold, dist≥24px from existing nodes, outside covered mask, edge_class≠0/16, not near end-sign (505,505)
4. If valid candidate found: place it as a seed node, continue autoregressive decoding from there with isolated component_preds (no cross-component last_edges)
5. Discard component if <3 points; repeat until cap or no valid start

**Component colors (`graph_overlay_components.png`):**
0=red/blue, 1=green, 2=purple, 3=orange, 4=cyan, discarded=gray.

**Setting matrix (3 settings):**

| Setting | restart_thr | min_dist | max_starts | dilation |
|---------|------------|----------|------------|----------|
| multistart_tuned_restart010_dist24_cap4 | 0.10 | 24 | 4 | 12 |
| multistart_tuned_restart015_dist24_cap4 | 0.15 | 24 | 4 | 12 |
| baseline_tuned_single_start             | —    | —  | 0 | —  |

**Results (5 samples, all "normal" category):**
- All 5 samples: restart seeds placed, but every restart component discarded (<3 pts)
- Root cause: isolated seed node fails `last_edges` direction constraint check in `_assemble_step`;
  model was trained for top-left-origin traversal, not mid-graph seeding
- avg_comp=1.0 and avg_pts identical to single-start baseline for all settings

**Recommendation:** Keep multi-start **OFF** for now.

**Next steps if multi-start is revisited:**
1. Skip `last_edges` constraint for the first decoding step after a restart seed, OR
2. Two-stage approach: mask covered regions on the image canvas and rerun normal inference from scratch
